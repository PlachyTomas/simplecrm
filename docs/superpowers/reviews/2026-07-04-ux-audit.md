# UX weak-point audit — SimpleCRM app

Scope: the whole app frontend (`frontend/src`), focused on how it *feels to
use* rather than correctness (that was the separate code review). Three
dimensions scanned: loading/error/empty states; forms/feedback/destructive
actions; accessibility/mobile/navigation.

Severity: **P1** breaks or badly confuses a real workflow (crash, silent
failure, blank screen, no confirm on a destructive action, unusable on mobile /
by keyboard) · **P2** noticeably worse than it should be (no skeleton, generic
error, weak empty state) · **P3** polish.

## 1. Loading / error / empty states

### P1

- [P1] deals/DealDetailPage.tsx:188 + useDealActions.ts:10 — **"Označit jako
  vyhráno" fails silently.** `markWon.mutate()` has no `onError` (nor does the
  hook). A failed POST just re-enables the button — no toast, no message — so
  the user can't tell if it registered. Fix: add an `onError` toast like the
  sibling reopen/delete/save handlers already have.
- [P1] deals/DealDetailPage.tsx:422 — **"Označit jako prohráno" fails
  silently.** `markLost.mutate` has `onSuccess` but no `onError`; a failure
  clears the spinner with zero feedback. Fix: `onError` toast, keep dialog open.
- [P1] calendar/CalendarPage.tsx:118 — **Calendar load-failure looks like an
  empty month.** The query's `isError` is never read, so a failed fetch renders
  "Žádné události…" — the user thinks their meetings vanished. Fix: read
  `isError`, show an error/retry banner distinct from the empty state.

### P2

- [P2] companies/CompaniesListPage.tsx:678 — **No loading indicator on first
  load** — the empty-state guard requires `!isPending`, so it shows an empty
  table with headers and no spinner while fetching (Deals/Pipeline/Contacts
  show "Načítání…"). Fix: skeleton/placeholder when `isPending`.
- [P2] pipeline/PipelinePage.tsx:709 + useBoard.ts:81 — **Drag-to-move
  silently snaps back on error** — optimistic update rolls back with no toast,
  so a rejected move is indistinguishable from a mis-drop. Fix: `onError` toast.
- [P2] deals/DealsListPage.tsx:28 — secondary data (stage/owner/company names)
  isn't gated or error-checked; cells render "—" then pop in, and stay "—"
  silently if those queries error. Fix: light skeleton or deliberate fallback.
- [P2] dashboard/DashboardPage.tsx:155 — plain "Načítání přehledu…" text (no
  KPI skeleton) so the grid pops in; error state has no retry. Fix: card
  skeletons + retry.

### P3

- [P3] settings/sections/IntegrationsSection.tsx:56 — Google "Propojit znovu"
  has no `onError` (first-time connect at :81 does) → silent failed reconnect.
- [P3] reports/ReportsPage.tsx:210 — a hard `config.isError` renders the
  skeleton forever (infinite skeleton). Fix: branch on error + retry.
- [P3] dashboard/DashboardPage.tsx:60,111 — manager widgets show "Nelze
  načíst." with no retry (Reports widgets offer one). Minor inconsistency.

**Handled well:** Reports widgets (exemplary skeleton/error-with-retry/empty
triad), Companies & Contacts list empty/error states, Pipeline top-level,
Company/Deal detail tabs, Billing cards, Auth login. The core gaps are
concentrated in Deal-detail mutations and the Calendar error path.

## 2. Forms / feedback / destructive actions

### P1

- [P1] settings/UsersSection.tsx:146 — **Deactivating a user has no
  confirmation.** The Aktivní/Deaktivovaný toggle mutates immediately on click,
  revoking a colleague's login with one stray click and no undo (the *cancel-
  scheduled-deactivation* path at :210 does confirm — inconsistent). The role
  `<select>` (:72) also commits instantly, including admin promote/demote. Fix:
  confirm dialog for deactivation (and ideally admin role changes).
- [P1] contacts/ContactDetailPanel.tsx:52 — **Assign-company fails silently.**
  `handleAssignCompany` swallows the error in an empty `catch`, and
  `updateContact.isError` is never rendered anywhere — a failed assignment
  leaves edit mode with no feedback, so the user assumes it saved. Fix: inline
  error or toast in the catch (AddContactModal already renders `isError`).

### P2

- [P2] settings/TeamsSection.tsx:109,119,131 — **API errors render as
  `[object Object]`.** `String(err.body)` stringifies the parsed JSON body.
  Fix: use `err.body.detail` like UsersSection/PipelineSection do.
- [P2] deals/AddDealModal.tsx:234 (and DealDetailPage.tsx:114) — **Invalid deal
  value is a silent no-op** — submit `return`s on NaN/negative with no message
  or field highlight; the user clicks "Uložit" and nothing happens. Fix: set an
  inline error on the value field.
- [P2] companies/bulk-email/BulkEmailWizard.tsx:176 — **Bulk send has only a
  review-step gate**, no explicit final "are you sure" for emailing N external
  recipients, and no way to preview the `{firma}/{kontakt}` substitution. (Send
  is correctly disabled during pending; 422 vs generic errors handled.) Fix:
  stronger final confirm + a preview of the rendered merge fields.

### P3

- [P3] deals/DealDetailPage.tsx:186 — "Označit jako vyhráno" closes the deal
  with no confirm while "prohráno" opens a dialog — asymmetric.
- [P3] settings/sections/PipelineSection.tsx:237 / TeamsSection.tsx:114 —
  delete-stage / delete-team confirms are generic and don't warn what happens
  to the deals/members attached.
- [P3] bulk-email step-2 "Další" is disabled with no hint why (AddDealModal's
  `missingSummary` line is the pattern to copy).
- [P3] Inline field validation is largely absent app-wide (modals rely on
  native `required` + disabled submit). Auth pages hand-roll messages — the
  stronger pattern to standardize on.
- [P3] Some settings errors surface raw backend `detail` (technical/backend
  voice) rather than the app's voice — acceptable but worth standardizing.

**Handled exceptionally well (reference patterns):** GDPR erase (type-the-name
confirm), cancel-subscription (two-step inline confirm), delete-deal (admin-
gated, names the deal, "nelze vrátit"), seat-downsize (pick exactly N users,
live cost delta), all Auth flows (mapped error codes → Czech copy, 429
handling), the import wizard (mandatory dry-run), and consistent double-submit
protection everywhere. There is no delete-company feature (companies are freed
to the pool), so no unconfirmed-deletion risk there.

## 3. Accessibility / mobile / navigation

### P1

- [P1] **In-app create/edit modals have no focus management and no Escape-to-
  close.** They set `role=dialog`/`aria-modal` but on open focus stays on the
  trigger *behind* the dialog, focus isn't trapped (you can Tab out behind the
  overlay), and Escape does nothing — on the most-used flows. Affected:
  companies/AddCompanyModal.tsx:206, deals/AddDealModal.tsx:333,
  contacts/AddContactModal.tsx:67, events/EventFormModal.tsx:165,
  deals/MarkLostDialog.tsx:43, pipeline delete-deal dialog (PipelinePage.tsx:
  1007), settings BillingSection.tsx:736, and the admin modals. Who: keyboard +
  screen-reader users. **The correct pattern already exists** in
  marketing/LandingPage.tsx MobileDrawer (focus-in, focus-restore, Escape) and
  partially in PrivacySection (autoFocus) / BulkEmailWizard (Escape). Fix:
  extract a shared `<Modal>` with focus trap + initial focus + Escape and adopt
  it everywhere.

### P2

- [P2] **Settings tables overflow on mobile with no horizontal scroll** —
  `w-full`/`min-w-full` tables with no `overflow-x-auto` wrapper: UsersSection.
  tsx:258, InvitationsSection.tsx:269, BlockedCompaniesSection.tsx:171,
  TeamsSection.tsx:188, and bulk-email/EmailCampaignsPage.tsx:107. Reachable on
  phones (More → Nastavení); trailing columns/controls become unreachable. Fix:
  wrap each `<table>` in `<div className="overflow-x-auto">` (PermissionsSection
  and PrivacySection already do exactly this).

### P3

- [P3] Settings table headers use bare `<th>` without `scope="col"`
  (DealsListPage sets it) — screen-reader column association.
- [P3] AppShell.tsx:139 — the signed-in avatar is `alt=""` and the adjacent
  name/email is `hidden md:block`, so on mobile there's no accessible indication
  of who's logged in. Fix: `alt={user.name}` (or expose the name to AT).

**Handled well:** a proper skip link (AppShell), responsive sidebar + mobile tab
bar with 44px targets, the landing mobile drawer (the gold-standard modal),
detail-page back links, broadly-labeled icon buttons, the companies list's
mobile-stacked list vs. desktop table (with `overflow-x-auto`), tab semantics on
detail pages, an `sr-only` data table for charts, and zero non-semantic
clickable divs. The app is above average on a11y/mobile — the one systemic gap
is the modals never adopting the focus/Escape pattern the team already wrote.

## Consolidated fix list (ranked)

**Do first (P1 — real workflow breakage):**
1. Deal detail: add `onError` toasts to mark-won and mark-lost (silent
   failures). — §1
2. Calendar: show an error/retry banner instead of an empty month on load
   failure. — §1
3. Users settings: confirm before deactivating a user (and admin role change).
   — §2
4. Contact detail: surface the assign-company error (currently swallowed). — §2
5. Extract a shared `<Modal>` (focus trap + initial focus + Escape) and adopt it
   across the create/edit modals. — §3 (biggest, most systemic; one component
   fixes ~10 dialogs)

**Then (P2 — noticeable friction):**
6. Pipeline drag-to-move: toast on rejected move (don't just snap back). — §1
7. Companies list: loading skeleton on first load. — §1
8. Teams settings: fix `[object Object]` error rendering (`err.body.detail`). §2
9. AddDeal / edit: inline error on invalid value instead of a silent no-op. §2
10. Wrap the overflowing settings tables in `overflow-x-auto`. — §3
11. Bulk email: stronger final send confirm + merge-field preview. — §2

**Polish (P3):** dashboard/reports/integrations error-retry consistency,
`scope="col"` on settings th, mobile avatar `alt`, symmetric win/lose confirm,
delete-stage/team warnings, inline-validation standardization.

**Theme:** the app already contains best-in-class versions of almost every
pattern it's missing (the landing modal, the reports widgets, the GDPR/cancel
confirms). Most fixes are *propagating an existing pattern*, not inventing one —
so a shared `<Modal>` and a couple of shared helpers would close most of this.
