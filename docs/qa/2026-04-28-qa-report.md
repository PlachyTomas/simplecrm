### QA-001 — [P0, SEC] Trial gate fails open: expired-trial org has full app and API access ✅ FIXED 2026-04-29

**Fix:**
- `backend/app/api/v1/__init__.py` — every protected router (organizations, companies, contacts, deals, pipelines, reports, teams, users) is now mounted with `dependencies=[Depends(require_active_trial_or_subscription)]`. The dependency itself was already implemented in `backend/app/core/deps.py` (and unit-tested in `backend/tests/services/test_permissions.py`) — it just hadn't been wired anywhere.
- `backend/app/api/v1/auth.py:me` switched from `Depends(get_current_user)` to `Depends(require_active_trial_or_subscription)`. `/auth/me` now 402s when the org's trial has ended without a `stripe_customer_id` — that's what `ProtectedRoute` listens for to render `<TrialExpiredGate />`.
- `/auth/logout` and `/auth/refresh` intentionally bypass: users must be able to sign out + re-hydrate their session even on an expired trial (the gate then takes over the next time `/auth/me` runs).

**Verification:**
- API curl with the expired-org JWT: `/auth/me` → 402 with `{detail: {detail: "Trial expired", trial_ends_at: …, organization_id: …}}`; `/companies`, `/deals`, `/reports/leaderboard` → 402; `/auth/logout` → 204. With the active-trial JWT: `/auth/me` → 200, `/companies` → 200.
- Browser: dev-login as `qa-expired@expired-org.cz` → redirected to `/app` → `<TrialExpiredGate />` renders the full-screen "Vaše zkušební doba skončila" overlay with the formatted CZK price, the "Přejít na předplatné" + "Exportovat data" buttons, and the support-email link. Page title becomes "Zkušební doba skončila — SimpleCRM".
- Screenshot: `qa-artifacts/2026-04-29/snapshots/11-trial-gate-1280.png`.

**Polish landed alongside:**
- `usePageTitle("Zkušební doba skončila")` added to `TrialExpiredGate.tsx` so the browser tab title flips when the gate takes over.
- The gate's two action buttons (`Přejít na předplatné`, `Exportovat data`) used to be no-ops because `ProtectedRoute` rendered `<TrialExpiredGate payload={trial} />` without `onSubscribe` / `onExport` props. Now wired:
  - **Exportovat data** — calls a new ungated `/api/v1/reports/export-csv` (extracted into `app/api/v1/data_export.py` and mounted outside `PROTECTED_DEPS`; users must be able to walk away with their data even on an expired trial). Frontend `downloadDataExport(accessToken)` passes `from=1970-01-01&to=today` so the default this-month-only window on the reports endpoint doesn't truncate a long-tenured customer's history, then fetches the CSV and triggers a blob download named `simplecrm-export-{date}.csv`. Verified end-to-end: clicking on the gate page produces the file (header + rows), Excel-friendly UTF-8 BOM included. **Scope honesty:** this is the *deals* export only — companies and contacts are not yet exportable. The CSV header row is `id,name,stage,stage_type,value,currency,owner_user_id,company_id,expected_close_date,closed_at,lost_reason,created_at`. A comprehensive multi-entity export is tracked as a follow-up.
  - **Přejít na předplatné** — opens `mailto:podpora@simplecrm.cz?subject=Předplatné%20SimpleCRM` until the real Stripe checkout flow lands (separately tracked as a Tier 3 feature gap, see QA-010).
- `data_export.export_deals_csv` adds the `﻿` UTF-8 BOM that the existing reports endpoint was missing — directly resolves the brief §4.6 "CSV export Excel-roundtrip" check that earlier passes had flagged informally.


**Where:** All app routes for any user belonging to an org whose `trial_ends_at < now()`. Tested with `Expired-org` whose `trial_ends_at` was set to 7 days ago.
**Viewport:** 1280×800 (also reproduces at 390×844)
**Reproduction:**
1. Pick an org (here, `Expired-org`, id `0d62cbab-...`) and roll its trial back: `UPDATE organizations SET trial_ends_at = NOW() - INTERVAL '7 days' WHERE name='Expired-org';`
2. Dev-login as that user: `POST /api/v1/auth/dev-login {"email":"qa-expired@expired-org.cz"}`. JWT is minted normally.
3. Open the SPA, fill in the dev-login form. The user is routed to `/app` and sees the full dashboard.
4. Confirm at the API:
   - `GET /api/v1/auth/me` → **200** with full user object
   - `GET /api/v1/companies` → **200**
   - `GET /api/v1/deals` → **200**
   - `GET /api/v1/reports/leaderboard?days=30` → **200**
5. The only visible signal of "expired" is a top-of-app banner reading `Zkušební verze končí za 0 dnů — vyprší 21. dubna 2026` with an `Upgradovat na Plný` CTA — and a close-X that dismisses it.

### QA-006 — [P2, FN] Direct URL navigation to a protected route drops the in-memory access token ✅ FIXED 2026-04-29

**Fix (backend, `backend/app/api/v1/auth.py`):**
- New `POST /auth/refresh` reads the httponly `simplecrm_refresh` cookie, validates type+sub, looks up the user, mints a fresh access token, **rotates the refresh cookie value** (a new JWT is issued each refresh), returns `{access_token, user}`. Note: rotation here is *cookie value* rotation, not server-side invalidation — the previous JWT is still cryptographically valid until its TTL elapses (the JWT `jti` claim is reserved but not yet checked against a blocklist). True rotation-with-invalidation is a follow-up (see QA-024 below).
- `dev-login` now also sets the refresh cookie via the same `_set_refresh_cookie` helper that the OAuth callback uses, so the dev workflow benefits from refresh exactly the way real OAuth does.
- `/auth/refresh` deliberately **does not** apply `require_active_trial_or_subscription`: refreshing the access token must work even on an expired trial so the frontend can then 402 on `/auth/me` and render the gate. That's the right layering — gate the resources, not the auth re-handshake.

**Fix (frontend, `frontend/src/auth/AuthContext.tsx` + `ProtectedRoute.tsx`):**
- `AuthProvider` now exposes a `refreshSettled` flag. On mount, if no in-memory token is present, it kicks off `POST /auth/refresh`. While that's in flight, `refreshSettled === false`. On 2xx the token is set and `refreshSettled` flips true; on any error (including 401) the token stays null and `refreshSettled` flips true.
- `clearAuth()` (used by logout) marks `refreshSettled = true` so we don't immediately try to refresh again after the user explicitly signed out.
- `ProtectedRoute` renders a `Načítání…` placeholder while `!accessToken && !refreshSettled`, instead of bouncing to `/login`. Once settled, the existing path runs (redirect to login, render `TrialExpiredGate`, or render children).

  - New `refresh_tokens` table (allowlist of active `jti`s, one row per device) — multi-device-friendly. Insert on issue, swap on refresh, delete on logout. Needs a migration + ~30 lines of model/repository, plus tests for unknown-jti / replay / logout. Recommended path; ~half-day of work.
  - Denylist of recently-rotated `jti`s, TTL = original token expiry — also multi-device-friendly, but the denylist grows for refresh_token_ttl_days before pruning. Less elegant than allowlist.

**Reasoning:** Part B is P3 defense-in-depth — not exploitable on its own (a refresh-token leak is already bad regardless). Scope of the allowlist option (DB migration + four endpoints touched + ≥4 new tests) didn't match the priority without an explicit go-ahead. Surfacing as a decision point rather than guessing.

**Where (Part B):** `POST /api/v1/auth/refresh`, `services/auth.upsert_*`, `auth.google_callback`, `auth.dev_login`, `auth.logout`.
**Files involved (Part A, fixed):** `backend/app/api/v1/data_export.py`, `backend/tests/api/v1/test_reports.py`.

**Severity:** P3 — defense-in-depth; not exploitable on its own.

## Appendix A — QA report layout convention (proposed)

This pass uses:

- `docs/qa/YYYY-MM-DD-qa-report.md` — the durable, indexable, grep-friendly markdown.
