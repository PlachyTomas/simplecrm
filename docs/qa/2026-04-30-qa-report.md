## Headline findings

- **QA-026 (P1, FN)** — Win-deal action via the Pipeline card "Vyhráno" button does **not** invalidate the dashboard's KPI cache. After clicking Vyhráno on a 50 000 Kč deal and soft-navigating to `/app`, *Otevřené obchody*, *Hodnota pipeline*, *Vyhráno tento měsíc*, and *Výnosy tento měsíc* all show pre-win values until the page is hard-reloaded. Reproduces the user's smoking-gun example.
- **QA-027 (P1, FN)** — Closed-as-lost deals (the convention from `docs/manager-task.md`: `closed_at IS NOT NULL` + `lost_reason IS NOT NULL`, stage stays in its open-type column) are still rendered as cards in the open Pipeline columns *and* counted in the column-header "X obchodů · Y Kč" totals. The `P lost` test deal at 40 000 Kč inflates the *Nový lead* column header to "2 obchody · 240 000 Kč".
- **QA-028 (P0, FN)** — Add Contact form silently creates orphan contacts. The form has no Firma picker, the `company_id` ends up `NULL` in the DB, and the empty state explicitly tells users "Kontakty patří k firmám. Po přidání je uvidíte… na detailu firmy." That promise is broken on the very first save, and there's no way to associate a company afterward (Contact detail has no Firma field).
- **QA-029 (P1, FN)** — Add Firmu modal has no inline contact field, so the brief's "streamline data entry" goal isn't met. Creating a company *and* its first contact takes two modals on two separate routes — and because of QA-028 the second step doesn't even link them.
- **QA-030 (P1, FN)** — Company detail page has 4 stub tabs out of 5 (Kontakty, Obchody, Aktivita, Poznámky). All four either say "připravujeme" or render an empty state with no add affordance. None of these were on the deferred-feature list — the underlying data already exists (deals are filtered by `company_id` server-side, `Contact.company_id` is the FK, `Company.note` is a column) so the gap is rendering, not domain modelling.

### QA-026 — [P1, FN] Winning a deal in Pipeline does not refresh the Dashboard KPIs / Leaderboard until full page reload ✅ FIXED 2026-04-30

**Where:** `/app/pipeline` → click "Vyhráno" on any open deal card → soft-navigate to `/app`.
**Viewport:** 1280×800 (also reproduces at 390×844)
**Reproduction:**
1. Log in as admin. Note dashboard KPI values (e.g. *Otevřené obchody = 1, Hodnota pipeline = 50 000 Kč, Vyhráno tento měsíc = 4, Výnosy tento měsíc = 460 000 Kč*).
2. Click *Pipeline* in the sidebar.
3. Click the small "Vyhráno" button on an open deal card (here `QA reactivity test`, 50 000 Kč).
4. Click *Přehled* in the sidebar (soft route change, **no** browser reload).
5. KPI cards show **the same** "1 / 50 000 / 4 / 460 000" — *the win didn't propagate*.
6. Hard-refresh the page → KPIs jump to "0 / 0 / 5 / 510 000".

**Expected:** The win mutation invalidates `["reports", "kpi-summary"]` (and `["reports", "leaderboard"]`, `["reports", "team-leaderboard"]`, `["deals", …]`) so any KPI consumer re-fetches on its next mount.

**Root cause hypothesis:** `frontend/src/app/pipeline/...` (the win mutation hook) probably only invalidates pipeline / deals queries, not the reports cluster. `frontend/src/app/dashboard/useKpi.ts` uses default `staleTime`, so without an explicit invalidation it serves the cached snapshot for the React Query window.

**Severity:** P1 — first thing a real manager checks after closing a deal is the dashboard total. Stale numbers undermine trust in every subsequent screen.
**Bucket:** FN
**Artifact:** `qa-artifacts/2026-04-30/snapshots/qa-pipeline-1280.png`, `qa-artifacts/2026-04-30/snapshots/02-dashboard-1280-light.png` (from 2026-04-29 — same dashboard layout).
**Files likely involved:** the pipeline card's win-mutation hook (somewhere under `frontend/src/app/pipeline/`), `frontend/src/app/dashboard/useKpi.ts`, `frontend/src/app/reports/useReports.ts`. Suggested fix: a shared `invalidateDealReadModels(qc)` helper that hits keys `["deals"]`, `["reports", "kpi-summary"]`, `["reports", "leaderboard"]`, `["reports", "team-leaderboard"]`, `["reports", "loss-reasons"]`, `["reports", "pipeline-velocity"]`, `["reports", "my-summary"]`, `["pipeline"]`. Call it from every deal-write mutation (win / lose / move stage / edit / create / delete).

**Same root cause likely affects:**
- Dashboard *Leaderboard (30 dní)* widget after a win.
- Reports page *Žebříček týmů* / *Žebříček obchodníků* / *Důvody prohry* / *Rychlost* after any deal write.
- *Moje výsledky* salesperson rollup after a contact / deal write.
- Treat as one fix, not five.

### QA-027 — [P1, FN] Closed-as-lost deals appear inside open Pipeline columns and inflate the column-header totals ✅ FIXED 2026-04-30

**Where:** `/app/pipeline`
**Viewport:** 1280×800
**Reproduction:**
1. Pick any open-type stage (e.g. *Nový lead*).
2. Insert a deal in that stage with `closed_at` set and `lost_reason` set (this is exactly the "lost" convention from `docs/manager-task.md`: lost deals stay in their current stage, just with `closed_at` + `lost_reason`).
3. Visit `/app/pipeline`. The lost deal still renders as a card in the *Nový lead* column.
4. The column header reads "2 obchody · 240 000 Kč" — including the lost 40 000 Kč.

**Expected:** Pipeline lists deals where `closed_at IS NULL`. Closed deals (won OR lost) belong on Reports / Deals list, not in the active funnel. Column totals ditto.

**Root cause hypothesis:** The deal-by-stage query that powers Pipeline doesn't filter on `Deal.closed_at IS NULL`. The closed-out path was probably written assuming "won deals are in the *Vyhráno* stage" (which is true, because of `stage_type=won`), but it forgot the symmetric "lost deals stay in their original stage" case.

**Severity:** P1 — visibly wrong totals on the most-used screen.
**Bucket:** FN
**Artifact:** `qa-artifacts/2026-04-30/snapshots/qa-pipeline-1280.png`
**Files likely involved:** `backend/app/api/v1/deals.py` (list endpoint with `?stage_id=…`) — add `Deal.closed_at.is_(None)` to the filter, or expose a `?closed=false` parameter that the pipeline view passes; OR fix on the frontend by filtering `deals.filter(d => d.closed_at == null)` before grouping. Backend filter is cleaner.
