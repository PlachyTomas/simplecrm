# Task 1.1 ‚Äî Organization / User / Team / Plan models + migration

## Goal
Lay down the four foundational tenancy and billing tables so subsequent auth
work (Task 1.2+) and data tables (Phase 2) have somewhere to hang. Seed the
default Plan rows so the billing summary endpoint in Phase 7 has something to
read from day one.

## Design decisions
- **UUID primary keys** (Postgres native `uuid` type + `uuid.uuid4` default).
  CRM records get linked around externally (invites, shared filters, audit
  log); UUIDs are less leaky and avoid cross-org collision risk.
- **Enums stored as Postgres `enum` types.** This gives us DB-level validation
  plus clean OpenAPI schemas. Enums created in the migration with
  `sa.Enum(..., create_type=True)` (default).
- **`trial_ends_at` is a non-nullable `timestamptz`**, set to
  `created_at + 30 days` at insert time. Real default applied in Python (server
  default in Postgres too, via a function call in the migration).
- **Team ‚Üí Manager**: `Team.manager_user_id` is nullable ‚Äî a team can exist
  before its manager is elected; deferrable FK to avoid chicken-and-egg issues
  when creating the org's first user inside a transaction.
- **`User.team_id` nullable**: salespeople may be unassigned; admins typically are.
- **Region / locale / currency**: `region` enum (seed with `"eu-cz"`), `locale`
  varchar default `cs-CZ`, `currency` varchar(3) default `CZK`. All i18n-ready.
- **Indexes** from Section 7 of the brief: `Organization.trial_ends_at`,
  `User.email` (unique), `User.google_id` (unique nullable),
  `User.organization_id`, `Team.organization_id`.
- **`email` is unique globally** ‚Äî Google sign-in with the same email always
  lands on the same User, even if they left a different org. Edge-case for
  MVP: if a user is already in one org, a second Google login reuses the same
  User record; they cannot belong to two orgs. Document in code comment.

## Acceptance criteria
1. `uv run alembic upgrade head` creates `organizations`, `plans`, `users`,
   `teams`, three enum types, and inserts two plan rows.
2. `uv run alembic downgrade base` drops everything cleanly (no orphan enums).
3. `uv run alembic check` reports no pending autogenerate diffs.
4. `uv run pytest` ‚Äî all existing tests still pass PLUS the new model tests
   (‚â• 3 assertions).
5. `mypy app` clean in strict mode.
6. No raw `Kƒç`, no `value_czk` columns, no currency hardcoded in the app layer.
7. One commit: `feat(db): foundational org/user/team/plan tables + seed ‚Äî Task 1.1`.
